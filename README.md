# Aa
"Micro-langs"

Hypothetical research on domain-specific micro programming languages. 

Much in the same spirit as the original CSS decoupling from HTML.

- [Domains in Programming](docs/domains.md)

## Abstract 
This branch of metalang is focused on researching and exploring micro-languages to separate the various domains in programming, into smaller, subsets of language and syntax, designed to specific parts of code.

## Inspiration 
The idea comes from CSS, and how it fundamentally --feels-- like a different language than HTML. It fundamentally solves a problem kmown as __["Ambiguous domains"]()__

## Significance of languages in human life 
The names we give to things changes the way we think about them, and the languages we use, changes the way we think. 

Because programming is fundamentally a language-based medium, it matters at the substrate-level, how we think and express in code. 

## Why separate domains?
Isn't the power of the language, the ability to express things in-line? 

This makes sense at surface level, however, in real code, it is incredibly difficult to distinguish between markup and source; between the program and the layers of security. This is usually the reason we get lost in the system, (e.g. missilanguagesrest for the trees.)


Significance of domain separation, could help us understand how these parts work together cohesively in a final compiled runtime. 



## Domain vs. Part
What classifies as a __domain__ _(worth separating)_, and a __part__ of programming?

- `while` loops are an example of part
- __Logic__ is an example of domain

Both are cohesively similar... _(dealing with rules and constraints)_... but what distinguishes them is subject; how do we group things, as human beings? [(@citation-needed)](#)

## Proposition 
arguably, the only reason we keep these domains conjoined into what Frankenstein language, is because we have access to variables within the same complex. separating these domains into separate micro languages, would introduce an enormous amount of design complexity. 

- Syntaxes
- File extension handlings for IDEs
- Unity of runtime model (Shared environment variables, like how CSS can inspect HTML without requiring direct in-line memory access)
- Compilation to JS (CJS) vs. Realtime interpretation (RTI)

## Conclusion 

No model is perfect. All systems are fundamental trade-offs. The trade-off of separation is always fragmentation of continuity, which is the integral of programming. 

Whether it is worth applying energy to this, or focusing on other aspects of programming, is entirely up to participants of this conversation.
